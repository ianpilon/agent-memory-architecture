Architecture decision

Monolith vs split

Default to the two-agent split unless there's a reason not to.

Action agent (the one users interact with)
[model, e.g. Qwen 3.5 9B local, or Claude / GPT-4 via API]
Memory agent (separate sidecar, runs every N steps, decides whether to intervene)
[smaller model, e.g. Qwen 3.5 9B or smaller]
Memory store
NapMem-style pyramid (raw → records → topic tracks → profile) + shared company knowledge layer above it
Intervention policy
selective — ship the silence

Why the split is the default:

  • Sizes the two models independently (cheap memory agent, strong action agent)
  • Works against any frontier action agent you can't retrain (closed models)
  • Cost lands on a bounded schedule, not every query — better under concurrency
  • The Proactive Memory Agent ablation shows silence beats always-on injection and passive full-bank exposure

Hardware verdict

Size for the memory, not the chip

ConfigFitsVerdict

M4 Mac mini, 24GB9B + modest concurrencyOK for 1–3 light users; prototyping
M4 Pro Mac mini, 48–64GB9B + real concurrency + headroomRecommended for client deploy
Cloud GPUsTraining only (GRPO)Rent, don't own

Rule of thumb: max the RAM, it's soldered. Memory is the bottleneck, not CPU/GPU. Navigation/intervention is 2–4× the inference calls of single-shot, so effective throughput is divided — size for that, not raw model speed.

RL training

Do we need it?

○  No — prompted navigation + intervention only ●  Yes — GRPO-trained policy
  • No — prompted navigation + prompted intervention only. Cheapest, ships fast, still beats passive retrieval. Right for most knowledge-retrieval use cases. Runs on the mini.
  • Yes — GRPO-trained navigation or intervention policy. Needed for persona/preference-heavy or high-precision cases. Requires training compute (cloud GPUs), then deploy the trained weights back to the mini. Check whether NapMem/Proactive released weights — if not, "follow the paper" means re-running their recipe.
What neither paper covers

Scope these explicitly

These are where real client projects fail, and where your billable judgment lives:

  • Cost & latency. Navigation/intervention = multiple LLM calls per query. Price it. Set the N-step cap.
  • Privacy & forgetting. Deletion, redaction, retention policy. Required for any real product.
  • Domain eval. The benchmarks (PersonaMem, tau-2-Bench) are synthetic. Build a domain-specific eval or you can't claim the gains transfer.
  • The closed-model RL problem. If the client is on OpenAI/Anthropic, you can prompt-navigate but can't GRPO-train their model. You lose the RL uplift unless you host a trainable model.
  • Ops. Auto-start on power loss, restart on crash, monitoring, backups (the memory pyramids are business data), updates without downtime.
